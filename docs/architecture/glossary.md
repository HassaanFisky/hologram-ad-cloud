# Platform Glossary — HoloLED Cloud

This glossary defines the precise meaning of every domain term used in the codebase, APIs, and documentation.

When in doubt about naming, use these definitions. If two terms seem interchangeable, this glossary resolves the ambiguity.

---

## Domain Entities

### Organization
A paying tenant on the HoloLED Cloud platform. Typically an advertising operator, media company, or out-of-home advertising business. Organizations own devices, manage customers, upload media, and create campaigns.

In the codebase, this maps to the `Company` Prisma model (legacy internal name). The public API and UI surface use "organization" terminology.

> **Do not confuse with:** Company (internal DB term), Workspace (not yet implemented), Customer (a client of the organization, not the organization itself)

---

### User
A human account that belongs to one organization and holds one of the platform roles. Users authenticate with email and password and receive JWT access and refresh tokens.

**Roles:**
- `PLATFORM_ADMIN` — Full platform access across all organizations
- `COMPANY_ADMIN` — Full access within their organization
- `OPERATIONS_MANAGER` — Manages devices, media, schedules, and groups
- `INSTALLER` — Registers and pairs physical devices
- `CUSTOMER_VIEWER` — Read-only access to customer campaign data
- `FINANCE` — Access to billing and invoicing data

---

### Customer
A brand or advertiser served by an organization. Customers have campaigns, proof-of-play access, and billing status, but they do not manage devices directly.

> **Do not confuse with:** User (a user has platform auth and a role), Organization (the operator, not the advertiser)

---

### Device
A physical hologram or 3D LED display unit registered in the platform. Each device has:
- A unique serial number
- A pairing lifecycle (registered → paired → online)
- A heartbeat connection to report status
- MQTT subscription for receiving commands
- A sync manifest for content delivery

---

### Device Group
A logical grouping of devices. Schedules can target a group rather than individual devices. Groups are scoped to an organization.

---

### Pairing Code
A short-lived one-time credential generated when a device is registered. The installer provides this code to the physical device agent to authenticate the initial pairing exchange.

---

### Heartbeat
A periodic status report sent from the device agent to the API. Contains firmware version, agent version, disk usage, network status, and any recent errors. Used to maintain online/offline status and detect degraded devices.

---

### Media Asset
A file (video, image, or hologram format) uploaded by an operator. Media assets go through a lifecycle:

```
UPLOADING → PROCESSING → READY → FAILED
```

Ready assets can be added to playlists. PROCESSING involves transcoding, thumbnail generation, and compatibility validation.

> **Blocked:** Exact hologram format specifications (codec, resolution, bitrate) require vendor hardware documentation.

---

### Playlist
An ordered list of media assets. Playlists define the sequence and duration for content playback. A playlist is created by an operator and assigned to a schedule.

---

### Schedule
A time-bound assignment of a playlist to one or more devices or device groups. A schedule has:
- Start datetime
- End datetime
- Target (device or device group)
- Priority (for conflict resolution)

---

### Sync Manifest
A JSON document generated by the API and downloaded by the device agent. It tells the device which media files to download and in what order to play them for each active schedule window. The device can play offline from the last downloaded manifest.

---

### Campaign
A customer-scoped view of one or more schedules. Customers see their campaign status, scheduled locations, and proof-of-play data.

---

### Proof of Play
Evidence that a specific media asset was actually played on a specific device at a specific time. Used for billing, reporting, and advertiser verification.

> **Status:** Not yet implemented. Requires device-side playback logging and upload.

---

### OTA Release
An over-the-air firmware or agent software update package. OTA releases are uploaded by platform admins and assigned to device groups. The device agent downloads, validates (checksum), and applies the update.

> **Blocked:** Actual firmware flashing requires vendor hardware documentation.

---

### Device Agent
The software process running on the physical hologram hardware. It handles heartbeats, manifest downloads, media caching, command listening, and OTA application. It communicates with the platform via HTTPS and MQTT.

---

### Gateway
The reverse proxy layer (`apps/gateway`) that sits in front of the API. It handles CORS at the edge, rate limiting, and routing to internal services.

---

### Worker
The background job processor (`apps/worker`). It handles async work: media optimization, thumbnail generation, schedule manifest compilation, device offline detection, notification dispatch, and analytics aggregation.

---

### Audit Log
An immutable record of all sensitive actions performed on the platform, including who did it, when, and what entity was affected. Required for security compliance and debugging.

---

### MQTT Topic Structure

```
devices/{deviceId}/commands     Cloud → Device (commands)
devices/{deviceId}/telemetry    Device → Cloud (telemetry)
devices/{deviceId}/acks         Device → Cloud (command acks)
```

## Status Enumerations

### Device Status
- `OFFLINE` — Device has not sent a heartbeat within the expected window
- `ONLINE` — Device is connected and reporting normally
- `DEGRADED` — Device is reporting with errors in its heartbeat
- `MAINTENANCE` — Device has been manually placed in maintenance mode

### Media Status
- `UPLOADING` — Client is uploading the file to object storage
- `PROCESSING` — Platform is processing the file (transcoding, thumbnail)
- `READY` — File is ready for playlist assignment
- `FAILED` — Processing failed; operator must re-upload

### OTA Status
- `DRAFT` — Release is being prepared, not yet assigned
- `ACTIVE` — Release is assigned to device groups
- `DEPLOYED` — Release has been applied to all target devices
- `DEPRECATED` — Release has been superseded

---

## Naming Clarification: Company vs. Organization vs. Workspace

| Term | Meaning | Where Used |
|---|---|---|
| `Company` | Internal Prisma model name (legacy) | Database, internal TypeScript types |
| `Organization` | Public-facing API and UI term | API routes, documentation, UI labels |
| `Workspace` | Not yet implemented. Planned as a subdivision within an organization | Future |

**Engineering rule:** New code should use "organization" in route paths and UI labels. The Prisma model name `Company` is acceptable internally until a migration renames it.
